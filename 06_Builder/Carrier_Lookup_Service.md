# Carrier Lookup Service — автопроверка контрагента по ИНН

> **Статус**: Идея/черновик ТЗ. Реализация после MVP парсера тендеров.
> Открыто: 28.04.26 по запросу Романа.
>
> **Цель**: убрать у диспетчера ручную работу по сбору документов и проверке контрагентов в открытых базах. Один ввод ИНН → готовая карточка с риск-скорингом.

## Постановка задачи

### Проблема (сейчас)

Диспетчер при появлении нового контрагента (Модель B) вручную:
1. Звонит логисту в Москве, просит документы.
2. Открывает 7+ сайтов (egrul, pb, fssp, kad.arbitr, fedresurs, zakupki РНП, **ГосЛог**) и проверяет каждый по ИНН.
3. Складывает результаты в карточку УЛ.
4. Выставляет рекомендацию.

Это **30–60 минут на одного контрагента**. При потоке 20 новых контрагентов в месяц — потеря 10–20 часов диспетчерского времени.

### Решение (вижн)

```
[Диспетчер]
   ↓ вводит ИНН
[Carrier Lookup Service]
   ├─ Параллельно:
   │   ├─ Запрос документов у логиста (email + форма загрузки)
   │   └─ Парс открытых баз по ИНН
   │       ├─ ЕГРЮЛ (egrul.nalog.ru — открытое API)
   │       ├─ ФНС "Прозрачный бизнес" (pb.nalog.ru — парсинг)
   │       ├─ ФССП (открытый API)
   │       ├─ kad.arbitr.ru (парсинг)
   │       ├─ bankrot.fedresurs.ru (парсинг)
   │       ├─ РНП на zakupki.gov.ru (парсинг)
   │       └─ Реестр ГосЛог (ручная или через интеграцию когда появится API)
   ↓
[Pre-filled карточка контрагента]
   ├─ Все поля заполнены
   ├─ Подсветка красных флагов
   ├─ Риск-скоринг (Высокий / Средний / Низкий)
   └─ Кнопка "Передать бухгалтеру на финальную проверку"
```

## Функциональные требования

### F.1 Ввод
- ИНН (обязательно).
- Тип контрагента: АТП / Экспедитор / Не уверен (диспетчер выбирает).
- Логист, который привёл контрагента (выбор из списка для отправки запроса документов).

### F.2 Запрос документов у логиста
- Автогенерируется email на адрес логиста.
- В письме — список требуемых документов (зависит от типа контрагента).
- В письме — уникальная ссылка на форму загрузки сканов.
- При загрузке через форму документы автоматически попадают в карточку контрагента в УЛ.
- Напоминание логисту через 24 часа, если не загрузил.

### F.3 Открытые базы
- ЕГРЮЛ, ФНС, ФССП, kad.arbitr, fedresurs, РНП — параллельные запросы.
- Время выполнения: ≤30 секунд (асинхронно с возможностью посмотреть промежуточный результат).

### F.4 Сводка
- Все поля карточки заполнены (см. ЧАСТЬ C в `Carrier_Audit_Checklist.md`).
- Красные флаги — отдельным блоком сверху.
- Риск-скор — высокий/средний/низкий с обоснованием.

### F.5 Передача бухгалтеру
- Кнопка "Запросить финальное заключение бухгалтера".
- В карточке статус меняется на "На финальной проверке".
- Бухгалтер получает уведомление в её рабочем интерфейсе.

## Нефункциональные требования

- Развернуть на том же VPS, что и парсер тендеров (общий сервер, экономия).
- Доступ — через УЛ или отдельную страницу (решим на этапе UI).
- Логи всех проверок хранятся 3 года (для налоговой защиты).

## Источники данных — детально

| Источник | Доступ | Сложность | Примечания |
|---|---|---|---|
| ЕГРЮЛ | Открытое API egrul.nalog.ru | Низкая | JSON-ответ, легко парсится |
| ФНС "Прозрачный бизнес" | pb.nalog.ru, парсинг | Средняя | API закрыт, нужен парсер страниц |
| ФССП | Открытое API fssp.gov.ru | Низкая | JSON, лимиты по запросам |
| kad.arbitr.ru | Парсинг | Высокая | Антиботная защита, нужен Playwright |
| bankrot.fedresurs.ru | Парсинг | Средняя | Возможно есть рудиментарное API |
| РНП | zakupki.gov.ru, парсинг | Низкая | Публичный реестр |
| **ГосЛог** | **Госуслуги, ручная** | **Высокая** | **API ожидается во 2-й половине 2026, пока — ручная проверка через личный кабинет Госуслуг** |
| СПАРК / Контур.Фокус | Платное API | Низкая | 50–200K ₽/год, можно подключить позже |

## Технологический стек (предварительный)

- **Backend**: Python 3.10+, FastAPI / Flask.
- **Парсеры**: Playwright (для kad.arbitr и других с антиботом), requests (для API).
- **БД**: PostgreSQL или SQLite (для MVP).
- **Очередь**: Celery / RQ (для параллельных запросов).
- **Email-шаблоны**: Jinja2 + SMTP компании.
- **UI**: либо встроить в УЛ через API, либо отдельная мини-страница на FastAPI + HTMX.

## Этапы разработки

### Этап 1. MVP — только открытые базы (1 неделя)
- Ввод ИНН → ЕГРЮЛ + ФНС + ФССП + bankrot.
- Простая таблица результатов.
- Без формы для логиста.

### Этап 2. Полная цепочка (1 неделя)
- Подключить kad.arbitr, РНП.
- Сборка карточки контрагента в формате полей УЛ.
- Риск-скоринг.

### Этап 3. Запрос документов у логиста (1 неделя)
- Email-шаблоны.
- Форма загрузки.
- Связка с УЛ (через API УЛ или экспорт в УЛ).

### Этап 4. ГосЛог-интеграция (когда появится)
- Подписаться на новости ГосЛог об открытии API.
- При появлении — добавить как ещё один источник.
